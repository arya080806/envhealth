# 2026-06-14 参与者账户匹配审计脚本与数据修复方案

## 一、当前结论

本轮排查发现，用户编号不稳定和部分账号登录“不存在”不是单一前端问题，而是身份数据已经在本地 SQLite 与飞书多维表格之间发生分叉。

已确认的核心事实：

1. 当前后端登录只查本地 SQLite，不会从飞书反查参与者。
2. 登录匹配入口是 `research_login_or_register()`，当前按“姓名 + 性别”查 `users`，然后取该用户最新的 `hci_participants`。
3. 飞书参与者主表中存在本地库没有的参与者，例如 `20260612-1`、`20260612-2`。
4. 飞书参与者主表中存在同名重复编号，例如 Arya 同时出现 `0002`、`0124`、`0126`。
5. 飞书里 `本地用户ID`、`本地参与者ID` 出现跨记录复用，说明远端表混入过不同本地库或旧部署实例的数据。
6. 下游表如自由创作元素摘要字段结构基本正确，但已经继承了上游错误编号，例如 Arya 的下游记录同时带有 `0002` 和 `0124`。

因此修复顺序必须是：先审计、再确认 canonical 编号、再做数据修复，最后改登录匹配逻辑。不能直接按最新记录或飞书某一行覆盖本地，否则会继续扩散错误编号。

## 二、审计脚本

新增只读脚本：

```powershell
.\.venv\Scripts\python.exe tools\audit_identity_data.py `
  --json-out docs\identity_audit_latest.json `
  --md-out docs\identity_audit_latest.md
```

只查本地 SQLite：

```powershell
.\.venv\Scripts\python.exe tools\audit_identity_data.py --no-feishu
```

追加关注对象：

```powershell
.\.venv\Scripts\python.exe tools\audit_identity_data.py `
  --target Arya `
  --target 20260612-1 `
  --target 20260612-2 `
  --target 其他姓名
```

脚本特性：

- 只读，不会写 SQLite，不会更新或删除飞书记录。
- 本地侧扫描 `users`、`hci_participants`、`sessions`。
- 飞书侧扫描参与者主表和所有已配置的下游同步表。
- 输出 JSON 供程序处理，输出 Markdown 供人工核对。
- 重点识别重复姓名、重复姓名 + 性别、重复编号、空编号/空性别、本地 ID 冲突、下游表编号与参与者主表不一致。

## 四、数据修复方案

### 阶段 0：冻结与备份

执行前先完成：

1. 暂停线上飞书同步任务或停服务维护窗口。
2. 备份线上 SQLite：`data/healing.db`。
3. 导出飞书参与者主表和所有下游表为 Excel/CSV。
4. 运行 `tools/audit_identity_data.py` 保存一份修复前审计报告。

验收标准：

- 有本地数据库备份。
- 有飞书导出备份。
- 有 `identity_audit_before.json/md`。

### 阶段 1：确认 canonical 参与者清单

人工产出一张清单，格式建议：

| canonical_code | registered_name | gender | keep_remote_record_id | duplicate_record_ids | action |
| --- | --- | --- | --- | --- | --- |
| 0002 | Arya | 女 | recvlACkqO3dYX | recvmf6LSaWLvN, recvmj3rIxfuJV | merge_or_archive |
| 0127 | 20260612-1 | 女 | 待确认 | 待确认 | import_to_local |
| 0128 | 20260612-2 | 女 | recvmkN4nwjW3a | recvmk08Cu5qXs | merge_note_then_archive |

验收标准：

- 每个同名冲突都只有一个 canonical 编号。
- 每个空编号/空性别记录都有处理动作：补全、合并、归档或删除。

### 阶段 2：修复飞书参与者主表

按 canonical 清单执行：

1. 保留 canonical 行。
2. 把重复行的重要备注、人口学字段合并到 canonical 行。
3. 重复行不要直接硬删，优先加备注如：`已归档：重复参与者，canonical=0002`，并移到归档视图。
4. 无编号空行如果只保留备注，则合并备注后归档。

验收标准：

- 飞书参与者主表中 `参与者编号` 唯一。
- `登记姓名 + 性别` 不再出现无解释重复。
- 空编号、空姓名、空性别记录为 0，或全部带明确归档备注。

### 阶段 3：修复下游表编号

需要检查所有下游表：

- 会话摘要
- 自由创作元素摘要
- 灵感创想元素摘要
- 对话改造摘要
- 智能参数摘要
- 模式使用次数
- 作品数量汇总

对每条记录执行：

1. 如果 `参与者编号` 指向重复旧编号，改为 canonical 编号。
2. 如果 `登记姓名` 与 canonical 编号对应姓名不一致，修正登记姓名。
3. 如果下游记录确实属于另一个真实参与者，则不要合并，改正姓名而不是改编号。

验收标准：

- 审计脚本中各下游表 `identity_mismatches` 为 0。
- Arya 下游记录不再同时出现 `0002` 和 `0124/0126`，除非人工确认它们是不同真实参与者。

### 阶段 4：本地库回流缺失参与者

飞书 canonical 清单确认后，把本地缺失但需要登录的参与者导入 SQLite：

- `20260612-1`
- `20260612-2`
- 其他审计脚本发现的飞书存在、本地不存在的受试者

建议新增一个受控导入函数，不从飞书全量自动导入：

```python
import_verified_hci_participant(
    participant_code='0128',
    registered_name='20260612-2',
    gender='女',
    remote_record_id='recvmkN4nwjW3a',
)
```

导入规则：

1. `participant_code` 必须在本地 `users.participant_id` 和 `hci_participants.participant_code` 中都不存在。
2. 若同名同性已存在，必须停止并要求人工处理。
3. 导入后同时创建 `users` 和 `hci_participants`。
4. 将 `feishu_sync_jobs` 的 hci participant job 绑定到对应 `remote_record_id`，避免下一次同步插入新行。

验收标准：

- `20260612-1`、`20260612-2` 登录不再显示不存在。
- 本地 `users.participant_id` 与 `hci_participants.participant_code` 一致。

### 阶段 5：修复后端登录匹配

改造 `research_login_or_register()`：

1. 支持编号优先登录：如果输入包含编号，先按编号精确匹配。
2. 姓名 + 性别匹配到 0 条：提示不存在，可注册或联系研究员。
3. 姓名 + 性别匹配到 1 条：正常登录。
4. 姓名 + 性别匹配到多条：不要静默取最新，提示“存在多个编号，请输入参与者编号或联系研究员”。
5. 新建参与者时检查本地与飞书待同步队列，避免复用编号。

验收标准：

- Arya 登录稳定显示 canonical 编号。
- 同名重复时不会随机显示最新编号。
- 20260612 系列账号导入后可正常登录。

### 阶段 6：修复飞书同步 upsert 防护

当前飞书同步按唯一字段查找记录，但如果远端已经有重复唯一键，会被 lookup 字典覆盖，导致“更新哪一行”不可控。

建议修改：

1. `find_record_id()` 遇到同一唯一键多条记录时抛出错误，不继续静默选择。
2. 同步队列将错误写为 `failed`，错误信息包含重复 record_id。
3. 研究人员先清理重复行，再重试队列。

验收标准：

- 飞书表出现重复唯一键时，系统停止写入并给出明确错误。
- 不再出现一个编号被同步到多条远端行的情况。

## 六、当前建议

先不要让登录页面自动从飞书全量回流，因为远端参与者表当前包含重复和半空记录。正确顺序是：

1. 用审计脚本生成完整冲突列表。
2. 人工确认 canonical 编号。
3. 清理或归档飞书重复记录。
4. 受控导入本地缺失参与者。
5. 修改登录逻辑，禁止重复匹配时静默选择最新记录。
