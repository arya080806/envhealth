## 二、已经核查到的事实

### 1. 入口与应用形态

- `https://yuhealth.cn/` 可访问，返回 200。
- `/login`、`/camera` 等普通路径返回 404，符合 hash 路由 SPA 的表现；真实前端路由应类似 `/#/login`。
- `/api/status`、`/health`、`/status` 均返回 404，说明当前生产站点缺少可公开探测的健康检查接口。

### 2. 后端接口状态

以下接口在未登录状态下返回 401，说明后端服务存在并启用了鉴权：

- `/api/games/status`
- `/api/games/history`
- `/api/auth/profile`
- `/api/health/daily`
- `/api/health/report`
- `/api/checkin/status`
- `/api/research/stats`

这只能证明接口服务在响应，不能证明 `20260612-1`、`20260612-2` 的游戏数据已经写库或同步飞书。要确认这两名患者的真实记录，必须使用管理员/研究者账号或服务器数据库权限查询。

### 3. 前端性能风险

线上资源存在明显的首屏和页面切换性能风险：

| 资源 | 大小/耗时观察 | 风险 |
| --- | --- | --- |
| `/static/js/pages-games.js?v=39` | 约 476 KB，曾观察到约 1.7 秒下载 | 游戏页逻辑集中，iPad 和弱网下切换/打开模式容易慢 |
| `/static/js/pages-health.js?v=41` | 约 218 KB | 健康页与游戏页共同增加 SPA 负担 |
| `/static/js/pages-games.js?v=39`、`/static/js/pages-health.js?v=41`、`/static/css/app.css?v=7` | 响应头为 `no-cache, no-store, must-revalidate` | 带版本号的静态文件仍不缓存，患者每次打开都可能重新下载 |
| `/static/img/doctor.png`、`/static/img/doctor-avatar.png` | 约 321-466 KB | 首页/头像资源偏大，弱网和 iPad 首屏加载压力增加 |

这些现象与用户反馈的“切换页面卡顿”“点击 AI 生成之后卡顿”“预设场景后任意模式图片加载慢”“iPad 端网页无法显示或很慢”高度相关。

- `20260612-1`、`20260612-2` 的游戏数据是否已经在数据库中但未同步飞书。

## 四、建议的底层修复执行计划

下面计划按“先不丢数据，再能观测，再提速，再做体验修复”的顺序执行。拿到 `yuhealth.cn` 源码或服务器权限后，应从阶段 A 开始。

### 阶段 A：先建立生产可观测能力

目标：不要再出现“患者做完了，但不知道数据在哪里失败”的情况。

1. 增加只读健康检查接口，例如 `/api/status` 或 `/api/healthz`。
2. 健康检查至少返回：
   - 后端服务状态。
   - 数据库连接状态。
   - 飞书配置是否完整。
   - 飞书最近同步时间。
   - 同步队列 pending/failed 数量。
   - 当前部署版本和构建时间。
3. 增加管理员只读诊断接口：
   - 按患者账号查询最近游戏提交记录。
   - 按患者账号查询最近飞书同步状态。
   - 按时间范围列出失败同步任务。
4. 所有接口要加管理员鉴权，不能公开暴露患者隐私。

### 阶段 B：修复游戏数据落库与飞书同步链路

目标：游戏数据先可靠进入数据库，飞书失败也能补偿。

1. 检查 `/api/games/submit`：
   - 是否在所有游戏模式下都会调用。
   - 是否包含患者账号、游戏类型、场景、操作记录、得分/结果、时间戳。
   - 前端失败时是否会提示或本地暂存。
2. 数据库写入与飞书同步解耦：
   - 游戏提交先写本地数据库。
   - 再写入同步队列。
   - 飞书失败不影响患者继续实验。
3. 同步队列必须具备：
   - 幂等键，避免重复同步。
   - 失败原因分类。
   - 指数退避重试。
   - 手动重试。
   - 按患者账号补同步。
4. 飞书字段映射启动时校验：
   - app token 是否有效。
   - table ID 是否存在。
   - 字段名是否匹配。
   - 权限是否足够。
5. 对 `20260612-1`、`20260612-2` 做一次只读追溯：
   - 查数据库是否有游戏记录。
   - 查同步队列是否有失败记录。
   - 若数据库有、飞书无，则补同步。
   - 若数据库无，则需要从前端日志、服务端访问日志或患者端缓存判断提交是否曾发出。

### 阶段 C：修复静态资源缓存和页面卡顿

目标：先解决“慢”和“iPad 打不开/白屏”的底层原因。

1. 对带版本号的静态资源启用长期缓存：
   - `/static/js/*.js?v=版本`
   - `/static/css/*.css?v=版本`
   - `/static/img/*`
2. 推荐响应头：
   - `Cache-Control: public, max-age=31536000, immutable`
   - 每次发布通过文件名 hash 或版本号变更强制刷新。
3. 拆分 `pages-games.js`：
   - 游戏模式页面按路由懒加载。
   - AI 生成相关逻辑单独懒加载。
   - 图表、html2canvas 等非首屏依赖只在需要时加载。
4. 对图片资源做分层：
   - 缩略图用于列表和预设入口。
   - 中等图用于模式页面。
   - 原图只在必须编辑或导出时加载。
5. AI 生成按钮点击后：
   - 立即进入 loading 状态。
   - 防重复点击。
   - 长任务走后端任务队列。
   - 前端轮询只查任务状态，不阻塞页面主线程。

### 阶段 D：iPad Safari 专项兼容

目标：不只在桌面浏览器通过，要在真实 iPad 上稳定可用。

1. 用真实 iPad Safari 测试：
   - 首页。
   - 登录。
   - 预设场景。
   - 任意模式打开。
   - AI 生成。
   - 游戏提交。
   - 飞书同步状态。
2. 检查并修复：
   - JS 语法是否兼容目标 iOS Safari。
   - viewport、安全区、滚动容器高度。
   - 图片解码内存。
   - 长任务阻塞。
   - hash 路由刷新和返回按钮。
3. 建议增加一套 iPad viewport 自动化烟测，至少覆盖白屏、路由、主按钮点击和核心接口响应。

### 阶段 E：上线前回归和现场保障

目标：下一次患者实验前，把风险提前暴露出来。

1. 准备两个测试患者账号，固定用于演练。
2. 每次上线后执行：
   - 登录。
   - 完成一个预设场景。
   - 进入一个游戏模式。
   - 提交一次游戏结果。
   - 后台确认数据库记录。
   - 后台确认飞书同步。
3. 建立现场应急流程：
   - 飞书不同步时先确认数据库是否有记录。
   - 数据库有记录则补同步，不中断患者实验。
   - 数据库无记录则检查前端提交和后端访问日志。
